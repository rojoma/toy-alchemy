# GitHub共同開発の教科書 — 素人向け完全ガイド

このプロジェクトでチームとして共同開発するための基礎知識です。
非エンジニアメンバーも読める内容になっています。

---

## Part 1: そもそもなぜGit/GitHubを使うのか

### 🏠 家の工事を例に

あなたが家を建てていて、作業ルールが何もない場合：

```
月曜: Aさんが壁紙を赤に塗った
火曜: Bさんが知らずに同じ壁を青に塗り替えた
水曜: Cさんが「先週の黄色の壁紙に戻して」と言う
       → もう元に戻せない
```

これがコードで起きると仕事が止まります。だから以下のルールが欲しい：

1. **誰が・いつ・何を変えたか** が全部記録されている（履歴）
2. 作業前に **別の場所で試せる**（下書き）
3. 変更前に **他の人がチェックできる**（レビュー）
4. 失敗したら **簡単に巻き戻せる**（元に戻す）

Git と GitHub はこの4つを解決するツールです。

## Part 2: 3つの基本概念

### ① リポジトリ (Repository / Repo)
**「プロジェクトの全ファイルと全履歴を保管する倉庫」**

```
toy-alchemy リポジトリ = このプロジェクト全体
├── コードファイル
├── ドキュメント
└── 「誰がいつ何を変えたか」の完全な記録（全履歴）
```

GitHub上の場所: https://github.com/rojoma/toy-alchemy

---

### ② ブランチ (Branch)
**「工事中のコピー空間」**

プロジェクトの状態を **枝分かれさせて並行作業** できる仕組み。

```
main ブランチ (本番用・常に動く状態)
  │
  ●─────────●─────────●  ← ここが公開版
  │
  │  feature/new-feature ブランチ (daigo の工事場)
  └──●──●──●  ← ここで自由に試せる。失敗してもmainには影響なし
  │
  │  fix/typo ブランチ (mach の工事場)
  └──●──●  ← 別の人が別の場所で並行作業
```

**ポイント**:
- `main` は常に動く状態を保つ（神聖な本番）
- 作業は必ず **main から枝分かれさせた別ブランチ** で行う
- 完成したらその枝を main に **戻す（=マージ）**

---

### ③ コミット (Commit)
**「セーブポイント」**

ゲームのセーブに似ています。作業の節目で「ここまでの変更を記録」する。

```
「README.md の誤字を直した」 ← これが1コミット
「ボタンの色を青に変えた」 ← これも1コミット
「ボタンのサイズも調整」 ← これも1コミット
```

各コミットには：
- **誰が** 作ったか
- **いつ** 作ったか
- **どのファイルをどう変えたか**
- **何のために変えたか**（コミットメッセージ）

が全部残ります。**コミットメッセージは未来の自分とチームへの手紙**です。

## Part 3: Pull Request (PR) — 最重要概念

### 🛒 「この変更、mainに入れてください」というリクエスト

PRは **「自分のブランチの変更を、レビューしてから main に取り込んでほしい」という依頼** です。

### なぜPRが大事か

直接mainに変更を入れない理由：

```
❌ 直接main変更
  Shoが「このコード動くはず」と思ってmain変更
    → 実は1行typoがあった
    → Railway自動デプロイ開始
    → 本番サイトが落ちる
    → 全員が使えなくなる

✓ PR経由
  Shoが変更 → PR作成
    → daigoが「これtypoでは？」とコメント
    → Shoが修正
    → daigoが「OK」(Approve)
    → マージ → 本番に反映
```

PRは **「4つの目でチェックする仕組み」** と思ってください。

## Part 4: 実際の流れ（PR作成を例に）

### 登場人物
- Sho: コードを変更する人
- daigo / mach: レビューする人

### ストーリー

```
🎬 シーン1: Sho が作業開始
────────────────────────────
Sho「PRテンプレートとIssueテンプレートを整備したい」

$ git checkout main           ← まず最新のmainを取得
$ git pull                     ← 最新を同期
$ git checkout -b chore/team-collab-setup  ← 新しいブランチを作る

この時点で:
  main ブランチ = 公開されている最新版
  chore/team-collab-setup ブランチ = Shoの工事場（mainのコピー）


🎬 シーン2: Sho が作業＆セーブ
────────────────────────────
Shoが .github/ の中にテンプレファイルを作成

$ git add .github/ .gitignore docs/ ...   ← 変更対象を指定
$ git commit -m "Add team collaboration setup"  ← セーブポイント作成

この時点で、Shoのブランチにだけ変更が記録されている。
mainはまだ何も変わっていない。


🎬 シーン3: ブランチをGitHubに送る
────────────────────────────
$ git push -u origin chore/team-collab-setup

Shoのパソコンのブランチが GitHub にアップロードされた。
でもまだmainには反映されていない。


🎬 シーン4: PR作成
────────────────────────────
GitHub画面で「Create pull request」ボタン

PR #1「Add team collaboration setup」が作成される。
→ daigo と mach に自動通知


🎬 シーン5: レビュー
────────────────────────────
daigo がPRを開く
  → 「Files changed」タブで何が変わったか見る
  → 気になる行に「この文言ちょっと硬いかも」とコメント
  → Shoが修正して再push
  → daigoが「Approve」ボタン


🎬 シーン6: マージ
────────────────────────────
Sho が「Merge pull request」ボタンを押す

main ブランチ が更新される
  → Railway が自動で本番デプロイ
  → 全員のブラウザで新バージョンが使える
```

## Part 5: 非エンジニアの関わり方

### daigo / mach が コードを書かずに貢献できる方法

#### 🟢 A. Issue を立てる（最頻出）

**「気になったこと・変えてほしいこと」を文章で書くだけ**

1. https://github.com/rojoma/toy-alchemy/issues に行く
2. 「New issue」を押す
3. 用途に合ったテンプレを選ぶ：
   - 📝 セッションのフィードバック（AI先生の挙動がおかしい）
   - 👩‍🏫 Teacher/Skill調整（先生のキャラ変更依頼）
   - 🐛 バグ報告（エラーが出る）
   - ✨ 新機能リクエスト
4. テンプレの質問に答える

**Issueは「このタスクをやってほしい」の公式な記録** です。Slackと違い、流れないし、誰でも後から探せます。

---

#### 🟡 B. PRをレビューする

エンジニアが作ったPRの内容をチェックします。**コードが読めなくてもできる**:

| 見るべきポイント | 誰でもできる |
|---|---|
| タイトルが「何をしたか」明確か | ✓ |
| Summaryに意図が書かれているか | ✓ |
| スクリーンショットに違和感はないか | ✓ |
| ドキュメントの日本語がおかしくないか | ✓ |
| 動作確認のチェックリストが埋まっているか | ✓ |

気になったら「Files changed」の行に直接コメントできます。

Approveが分からない場合は、コメントで「ここが気になります」と伝えるだけでOK。

---

#### 🔵 C. GitHub Web UIで直接編集する

実は **非エンジニアでもテキストファイルならブラウザ上で編集可能** です。

例: `field/external_teachers/warm_v1.json` の warmth を 0.9 → 0.95 に変えたい

```
1. GitHubでそのファイルを開く
2. 右上の鉛筆マーク ✏ を押す
3. 直接編集
4. 一番下の「Commit changes」
   → 「新しいブランチを作ってPRを作る」を選択
5. 自動でPRができる
```

これはJSON/Markdown/テキストファイル限定で有効。**コードは触らない前提**です。

## Part 6: よく出てくる言葉の辞書

| 用語 | 意味 | 例え |
|---|---|---|
| リポジトリ (repo) | プロジェクト全体 | 家丸ごと |
| ブランチ (branch) | 作業用コピー空間 | 工事中の仮設部屋 |
| コミット (commit) | 変更のセーブポイント | ゲームのセーブ |
| プッシュ (push) | ローカル→GitHubに送る | 郵送する |
| プル (pull) | GitHub→ローカルに取り込む | 郵便を受け取る |
| PR (Pull Request) | mainへの取り込み依頼 | 決裁書 |
| マージ (merge) | ブランチを統合 | 工事完了で本体に接続 |
| レビュー (review) | PRの内容チェック | 上司の承認プロセス |
| Approve | 「OK」ボタン | 決裁の判子 |
| Conflict (衝突) | 同じ箇所を別々に変えて混乱 | 同じ壁を2人が別の色に |
| Issue | やること・質問 | タスクチケット |
| main | 公開中の本番状態 | 引き渡し済みの家 |

## Part 7: よくある混乱ポイント

### 「ブランチ」と「PR」って何が違うの？
- **ブランチ**: 工事空間（物理的なコピー）
- **PR**: その工事の承認申請書類（手続き）

工事してもPRを出さなければ永遠に本体に反映されない。逆にPRだけ作っても中身のブランチがなければ意味がない。

### 「コミット」と「プッシュ」って何が違うの？
- **コミット**: 自分のパソコンで変更を記録
- **プッシュ**: その記録をGitHubにアップロード

コミットだけだと他の人には見えない。プッシュして初めて共有される。

### PRが一度マージされたら後戻りできない？
できます。マージは ただのコミットなので、「そのコミットを取り消す」コミットを作れば戻せます（`git revert`）。ただしできれば事前にレビューで見つけたい。

## Part 8: チーム運用ルール（本プロジェクト固有）

1. **mainに直接pushしない**（Shoを除く緊急時のみ）
2. **必ずブランチを作ってPR経由で変更**
3. **PRは1人以上のApprove後にマージ**
4. **コミットメッセージは「何をしたか」を具体的に** (「更新」「修正」はNG)
5. **ブランチ名の命名規則**:
   - `feature/xxx` — 新機能
   - `fix/xxx` — バグ修正
   - `chore/xxx` — 雑務（ドキュメント、設定）
   - `prompt/xxx` — プロンプト調整
   - `docs/xxx` — ドキュメント追加・修正
6. **作業開始前に必ず `git pull` で main を最新にする**
7. **PRテンプレート/Issueテンプレートに従って書く**（空白で出さない）
